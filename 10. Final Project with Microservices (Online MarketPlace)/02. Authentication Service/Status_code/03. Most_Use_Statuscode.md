


## üåê What Are HTTP Status Codes?

HTTP status codes are **numeric signals sent by a server** to tell the client **what happened to a request**.

They answer questions like:

* Was the request successful?
* Was the request wrong?
* Is the server broken?
* Should the client try again?

# ‚úÖ **2xx ‚Äì Success Status Codes**

These indicate the request was **received, understood, and processed successfully**.

---

## **200 ‚Äì OK (Most Used Status Code)**

### Meaning

The request was **successful**, and the server returned the expected response.

### When to Use

Use **200** when:

* Data is fetched successfully
* An update succeeds
* An action completes successfully

### Examples

* `GET /users` ‚Üí list of users returned
* `PUT /profile` ‚Üí profile updated
* `POST /login` ‚Üí login successful

### Key Notes

* Default success response
* Response body usually contains data
* Do **not** use for resource creation

---

## **201 ‚Äì Created**

### Meaning

The request succeeded, and a **new resource was created**.

### When to Use

Use **201** when:

* A new record is created in the database
* Server generates a new resource ID

### Examples

* `POST /users` ‚Üí new user created
* `POST /orders` ‚Üí order created

### Key Notes

* Usually returned with a **Location header**
* Indicates permanent creation
* Better than 200 for POST-create operations

---

## **204 ‚Äì No Content**

### Meaning

The request succeeded, but there is **nothing to return**.

### When to Use

Use **204** when:

* A delete operation succeeds
* An update succeeds but no data is needed

### Examples

* `DELETE /users/5`
* `PUT /settings` (no response body)

### Key Notes

* No response body allowed
* Cleaner for simple success actions

---

# ‚ùå **4xx ‚Äì Client Error Status Codes**

These indicate the **client made a mistake**.

---

## **400 ‚Äì Bad Request**

### Meaning

The request is **invalid or malformed**.

### When to Use

Use **400** when:

* Required fields are missing
* Input validation fails
* JSON syntax is incorrect

### Examples

* Missing email or password
* Invalid date format

### Key Notes

* Client must fix the request
* Most common API error code

---

## **401 ‚Äì Unauthorized**

### Meaning

Authentication is **missing or invalid**.

### When to Use

Use **401** when:

* Token is missing
* Token is invalid or expired

### Examples

* Accessing protected API without login
* Invalid API key

### Key Notes

* Means **not logged in**
* Often returned with `WWW-Authenticate` header

---

## **403 ‚Äì Forbidden**

### Meaning

Client is authenticated but **not allowed** to access the resource.

### When to Use

Use **403** when:

* User lacks permissions
* Role-based access denied

### Examples

* Normal user accessing admin API

### Key Notes

* Authentication ‚â† authorization
* Do not confuse with 401

---

## **404 ‚Äì Not Found**

### Meaning

The requested resource **does not exist**.

### When to Use

Use **404** when:

* Resource ID is invalid
* Endpoint does not exist

### Examples

* `GET /users/9999`
* Typo in URL

### Key Notes

* Very common
* Safe to expose publicly

---

## **409 ‚Äì Conflict**

### Meaning

The request conflicts with the **current server state**.

### When to Use

Use **409** when:

* Duplicate entries
* Version conflicts
* Unique constraint violations

### Examples

* Registering with existing email
* Updating stale data

### Key Notes

* More precise than 400
* Indicates logical conflict

---

## **422 ‚Äì Unprocessable Entity**

### Meaning

The request format is correct, but **data is semantically invalid**.

### When to Use

Use **422** when:

* Validation rules fail
* Business logic validation fails

### Examples

* Password too weak
* Age below allowed limit

### Key Notes

* Popular in REST APIs
* More specific than 400

---

# üí• **5xx ‚Äì Server Error Status Codes**

These indicate the **server failed**, even though the request was valid.

---

## **500 ‚Äì Internal Server Error**

### Meaning

The server encountered an **unexpected error**.

### When to Use

Use **500** when:

* Unhandled exceptions occur
* Application crashes
* Database failure

### Examples

* Null pointer exception
* Service unavailable internally

### Key Notes

* Do not expose internal details
* Log internally for debugging

---

## **502 ‚Äì Bad Gateway**

### Meaning

Server received an **invalid response from another server**.

### When to Use

Use **502** when:

* Microservice fails
* Upstream API fails

---

## **503 ‚Äì Service Unavailable**

### Meaning

Server is **temporarily unavailable**.

### When to Use

Use **503** when:

* Server under maintenance
* Server overloaded
